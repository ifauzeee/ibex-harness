# Milestone 1.1.4: Token Creation and Management API

**Status:** Planned
**Goal:** [1.1 Persistence and auth data plane](../goals.md)
**Phase:** 1 — Core Platform

## Why This Milestone Exists

Milestone 1.1.3 implements `ValidateToken`. But to validate a token, a token must
first exist. The current plan has no milestone for creating tokens.

This creates a hidden dependency: every integration test in 1.1.3, 1.2.1, and
beyond must manually INSERT rows into the `tokens` table with a pre-computed
Argon2id hash. This is fragile, insecure (test code computing hashes), and
inconsistent with the architecture.

The auth service must expose a `CreateToken` RPC (gRPC) and/or a REST endpoint
for token creation. This is also required for the dashboard (Phase 3+) and for
the CLI (Phase 4+) to create PATs.

`DATABASE_SCHEMA.md` defines the full `tokens` table including `prefix`, `hash`,
`permissions`, `expires_at`. `ARCHITECTURE.md §5` describes PAT creation as
"never stored in plaintext after creation." This milestone implements that contract.

## Tasks

### 1. Extend `ibex.auth.v1` proto with token management RPCs

Add to `packages/proto/proto/ibex/auth/v1/auth.proto`:

```protobuf
// CreateToken creates a new token and returns the plaintext value ONCE.
// The plaintext is never stored; only the Argon2id hash is persisted.
// Callers must store the returned plaintext immediately.
rpc CreateToken(CreateTokenRequest) returns (CreateTokenResponse);

// RevokeToken marks a token as revoked. Subsequent ValidateToken calls
// for this token will return UNAUTHENTICATED.
rpc RevokeToken(RevokeTokenRequest) returns (RevokeTokenResponse);

// ListTokens returns token metadata (never plaintext) for an org.
rpc ListTokens(ListTokensRequest) returns (ListTokensResponse);

message CreateTokenRequest {
  string org_id = 1;
  string name = 2;
  string description = 3;
  TokenType type = 4;
  int64 permissions = 5;  // permission bitmap
  google.protobuf.Timestamp expires_at = 6;  // optional
}

message CreateTokenResponse {
  string token_id = 1;
  string plaintext = 2;  // ONLY returned here; never stored
  string prefix = 3;     // first 8 chars, safe to display
  google.protobuf.Timestamp created_at = 4;
}
```

### 2. Token generation algorithm

```go
// GenerateToken generates a cryptographically secure token with prefix.
// Format: ibex_{type_prefix}_{random_base62_32chars}
// Example: ibex_pat_7f3k2m9xQr8nPvLwYzBcDeFgHiJkMnOp
//
// The prefix (first 12 chars including "ibex_pat_") is stored in the DB
// for display purposes. The full token is hashed with Argon2id.
func GenerateToken(tokenType TokenType) (plaintext, prefix string, err error)
```

**Security requirements:**
- Use `crypto/rand` for random bytes — never `math/rand`
- Minimum 256 bits of entropy (32 bytes → base62 encoded)
- Token prefix format: `ibex_{type}_{first_8_chars}` (safe to display in UI)
- Full token: never logged, never returned after creation response

### 3. Argon2id hashing (coordinate with Milestone 1.1.6)

```go
// HashToken hashes a plaintext token using Argon2id.
// Parameters must match ADR-0007 (Argon2id parameters).
func HashToken(plaintext string) (hash string, err error)

// VerifyToken verifies a plaintext token against its stored hash.
// Uses constant-time comparison to prevent timing attacks.
func VerifyToken(plaintext, hash string) (bool, error)
```

### 4. Repository layer

```go
type TokenRepository interface {
    // CreateToken inserts a new token. Returns the token ID.
    // The hash is stored; plaintext is never passed to this layer.
    CreateToken(ctx context.Context, params CreateTokenParams) (uuid.UUID, error)

    // RevokeToken sets revoked_at on the token.
    // Returns ErrNotFound if token does not exist in this org.
    RevokeToken(ctx context.Context, orgID, tokenID uuid.UUID) error

    // ListTokens returns token metadata for an org (no hashes).
    ListTokens(ctx context.Context, orgID uuid.UUID,
        cursor string, limit int) ([]TokenMetadata, string, error)
}
```

**Critical:** The repository layer must NEVER accept or return plaintext tokens.
The service layer generates the plaintext, hashes it, and passes only the hash
to the repository.

### 5. Service layer

```go
type TokenService interface {
    // CreateToken generates a token, hashes it, persists the hash,
    // and returns the plaintext ONCE. The plaintext is not stored anywhere.
    CreateToken(ctx context.Context, req CreateTokenRequest) (CreateTokenResult, error)

    // RevokeToken revokes a token and triggers bloom filter invalidation.
    RevokeToken(ctx context.Context, orgID, tokenID uuid.UUID) error
}
```

### 6. gRPC handler

Wire `CreateToken`, `RevokeToken`, `ListTokens` RPCs to the service layer.

**Authorization:** Token creation requires the caller to have
`PERMISSION_TOKEN_CREATE` bit set. Token revocation requires
`PERMISSION_TOKEN_REVOKE` or ownership. This is enforced in the handler.

## Testing Requirements

- Unit: token generation entropy (verify 256+ bits), prefix format, hash/verify round-trip
- Unit: constant-time comparison (verify no timing difference between valid/invalid)
- Integration: create token → validate token → revoke token → validate returns UNAUTHENTICATED
- Security: plaintext never appears in logs (log capture test)
- Cross-tenant: org A cannot revoke org B's token

## Definition of Done

- [ ] `CreateToken` RPC returns plaintext exactly once; subsequent calls cannot retrieve it
- [ ] Token prefix stored in DB; safe to display in UI
- [ ] `RevokeToken` causes subsequent `ValidateToken` to return UNAUTHENTICATED
- [ ] No plaintext token in any log line (verified by log capture test)
- [ ] `testutil.SeedToken` uses the real creation path
- [ ] Integration tests pass with real Postgres

## Risks

| Risk | Mitigation |
|------|-----------|
| Timing attack on hash comparison | Use `subtle.ConstantTimeCompare`; test with timing analysis |
| Token entropy insufficient | Use `crypto/rand`; test with NIST SP 800-90B entropy estimate |
| Plaintext leaks in error messages | Review all error paths; never include token value in errors |
