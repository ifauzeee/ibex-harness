# Milestone 1.2.1: Proxy Auth Client and Middleware

**Status:** Planned  
**Goal:** [1.2 Proxy platform integration](../goals.md#goal-12-proxy-platform-integration)

## Summary

Add gRPC client from proxy to auth `ValidateToken`, HTTP middleware that enforces authentication on protected routes, and request context carrying `org_id` and permissions.

## Branch

`feature/m1-2-1-proxy-auth-client`

## PR title (example)

`feat(proxy): auth gRPC client (m1.2.1)`

## Estimated effort

3–4 days

## Prerequisites

- [1.1.3](1.1.3-auth-token-validation.md) merged
- Auth and proxy run locally against compose

## Tasks

- [ ] `services/proxy/internal/authclient/` — gRPC client with timeout (e.g., 50ms), connection config
- [ ] Middleware: extract bearer token; call ValidateToken; attach context; fail 401/403
- [ ] Map gRPC errors to stable HTTP error envelope per API_DOCUMENTATION.md
- [ ] Config: `AUTH_GRPC_ADDR`, timeouts
- [ ] Integration test: proxy + auth + postgres; valid/invalid token
- [ ] Cross-tenant: token from org A used on org B resource path → 403 or 404 per policy
- [ ] Metrics: auth middleware latency; do not label by org_id

## Testing requirements

- Unit: middleware with mocked auth client
- Integration: real auth service; no mock for gRPC contract behavior

## Definition of done

- [ ] Protected route returns 401 without `Authorization`
- [ ] Valid dev token returns 200 on health of protected probe route (or stub LLM route returning 501)
- [ ] Auth service down → 503 fail closed (not bypass)
- [ ] Timeouts enforced

| Risk | Mitigation |
| --- | --- |
| Circular dependency auth ↔ proxy | Auth never calls proxy |
