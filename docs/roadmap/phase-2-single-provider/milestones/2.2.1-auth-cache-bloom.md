# Milestone 2.2.1: Auth Cache and Bloom Filter (optional)

**Status:** Planned (optional)  
**Goal:** [2.2 Authenticated critical path](../goals.md#goal-22-authenticated-critical-path)  
**Phase:** 2 — Single Provider E2E

## Summary

Add the bloom filter → LRU → gRPC validation pipeline described in [ARCHITECTURE.md](../../../ARCHITECTURE.md) and deferred from Phase 1 ([ADR-0011](../../../adr/ADR-0011-proxy-auth-client.md) §7). Wrap `auth.TokenValidator` with a cache decorator; enforce revocation propagation within documented SLA.

## Prerequisites

- Phase 1 Goal 1.2 complete (proxy auth middleware on `main`)
- Redis available for bloom + validated-token keys per [DATABASE_SCHEMA.md](../../../DATABASE_SCHEMA.md)

## Scope

- Redis bloom filter for fast invalid-token rejection
- In-process LRU cache (TTL-bound) for validated claims
- `CachingValidator` implementing `auth.TokenValidator`
- Revocation invalidates cache entries (integration tests per TESTING_STRATEGY)
- Audit flag when serving from cache during auth degradation (if policy allows)

## Out of scope

- Replacing gRPC validation entirely
- Negative cache without bloom
- Org-scoped rate limiting (Phase 1.2.4 / Phase 4)

## Definition of done (draft)

- [ ] Cache hit path skips gRPC when claims are fresh and not revoked
- [ ] Revoked token rejected even if previously cached
- [ ] Bloom false-positive rate documented and tested
- [ ] No bearer tokens in logs or metrics labels
